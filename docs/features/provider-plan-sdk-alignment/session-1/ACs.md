# Session 1 — Acceptance Criteria

**Branch:** `feat/plan-scenario-harness` (proposed)
**Plan:** [`../plan.md`](../plan.md)
**Architecture:** [`../architecture.md`](../architecture.md)

> Each AC is a **category**. The actual test cases inside are verified by `npm run test:ci` (Layer 2/3/4), `npm run test:e2e` (Layer 5), and one-shot scripts for Layer 6. The AC table tracks Pass / Agent / QC / Reviewer at the category level. Individual assertions inside a test file are implementation details — they pass or fail collectively under the category AC.
>
> **How column convention:** every AC's How field is structured as **Presence / Absence / Placement** — what should appear, what should NOT appear where it shouldn't, and what structural slot it occupies. This format is what catches positional bugs (text in wrong card section, icon on wrong side of label, etc.) — see architecture.md §5 Layer 2.
>
> **Drift-injection ritual:** AC-DRIFT verifies each layer can detect a synthetic regression. Run once per layer before AC closes, document the injection + failure mode in the QC column, then revert. Without this, tests can be decorative — passing while not actually able to fail.
>
> **Ship-gating vs. coverage:** Today's priority is shipping hosted/trial. ACs flagged **GATING** must pass for ship; ACs flagged **COVERAGE** are part of the harness but not required for ship if their scenarios (self-hosted dev keys) aren't seeded yet. See architecture.md §9 for the customer-reality rationale.

## Review checklist before approval

Before this AC doc moves into implementation, verify:

- [ ] **Coverage map.** Every required field in `HydratedPlan`, `PlanState`, `UsagePool`, `PlanAction`, `ConfirmActionConfig` is touched by at least one AC (mostly AC-CT-COMMON; some state-unique fields in AC-CT-{NONE,ACTIVE,TRIAL,EXPIRED,CANCELLED,INACTIVE}; modal fields in AC-CT-MODAL).
- [ ] **Optional fields.** Each optional field has Presence and Absence assertions in its AC's How column.
- [ ] **Positional ACs.** Where layout matters (iconBefore/After ordering, "Ended on" subtitle in header not body, deprovision date in header), Placement assertions are explicit in the How column.
- [ ] **Provider concerns out of scope.** Plan-business-spec validation (Priority Support = 5 tickets), cadence enforcement (monthly replenishment), and pool credit accounting are documented as provider-owned in `architecture.md` §10; no store-side AC enforces these.
- [ ] **Cross-repo enhancements filed.** SDK-RFC-1 (cadence field), SDK-RFC-2 (pools on every state), PLAT-1 (plan spec validation), PLAT-2 (cadence enforcement) are listed in `plan.md` cross-repo coordination section.
- [ ] **Drift-injection.** AC-DRIFT has one injection per testable layer (2, 3, 4, 5).
- [ ] **No spec validation in store.** "Priority Support has 5 tickets" is not an AC; the store renders whatever the provider sends.
